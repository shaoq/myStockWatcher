## Context

当前每日报告页面包含多个组件：
1. 概览卡片（4个统计指标）
2. 达标率卡片（达标率 + 较昨日变化）
3. 达标个股列表
4. 未达标个股列表
5. 趋势图表（近7日柱状图）

用户反馈表明，核心关注点在于查看哪些股票达标/未达标，统计卡片和趋势图使用频率低，且增加了页面复杂度和加载时间。

## Goals / Non-Goals

**Goals:**
- 简化页面布局，只保留核心的达标/未达标个股列表
- 减少不必要的API调用（移除趋势数据请求）
- 减少代码复杂度，提升可维护性
- 保持日期导航、交易日状态等辅助功能

**Non-Goals:**
- 不改变达标/未达标个股的展示逻辑
- 不改变后端报告数据结构（summary 仍然计算，只是前端不展示）
- 不改变分页逻辑

## Decisions

### Decision 1: 保留后端 summary 计算

**选择**: 后端 `get_daily_report()` 函数保持不变，继续计算 summary 数据

**原因**:
- summary 数据可能在其他地方使用（如API调用者）
- 后端计算成本低，移除可能影响现有API使用者
- 前端选择不展示即可，无需破坏性修改

**替代方案**: 移除后端 summary 计算 → 被拒绝，因为会破坏API响应结构

### Decision 2: 完全移除趋势API端点

**选择**: 从后端移除 `/reports/trend` 端点及相关代码

**原因**:
- 趋势功能不再使用，保留死代码增加维护负担
- 减少不必要的代码路径
- 如果未来需要，可以从git历史恢复

### Decision 3: 保持组件文件结构不变

**选择**: 在现有 `DailyReport.jsx` 中修改，不拆分新组件

**原因**:
- 修改量适中（移除约200行代码），不需要重构
- 保持现有文件结构，减少变更范围

## Risks / Trade-offs

### Risk: 用户可能需要统计数据

**缓解**:
- 达标/未达标数量在卡片标题中显示（如"达标个股 (12只)"）
- 如果用户强烈需要，可以快速回滚或添加新功能

### Risk: API响应包含summary但前端不使用

**缓解**:
- 不影响功能，只是响应体积略大
- 未来可以按需清理

## Migration Plan

1. 前端移除趋势数据获取和相关渲染代码
2. 前端移除概览卡片和达标率卡片渲染代码
3. 前端移除 `getTrendData` API 方法
4. 后端移除 `/reports/trend` 端点
5. 后端移除 `get_trend_data()` 函数
6. 后端移除 `TrendDataPoint` 和 `TrendDataResponse` schema

无需数据库迁移，无需特殊部署策略。
